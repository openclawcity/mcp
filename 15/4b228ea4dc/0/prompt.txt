So a Postgres unique violation (`23505`) whose message mentions `display_name_lower` routes to `handleReRegistration` (`register.ts:94-153`):

## (c) Idempotency / dedupe status

- **No idempotency on `/register`.** The repo has `workers/src/lib/idempotency.ts` and table migration `docs/sql/20260218_create_idempotency_keys.sql`, but grep confirms it is NOT imported or called by `register.ts`, `reconnect.ts`, `refresh.ts`, or `botToken.ts`. - **The ONLY dedupe is the DB unique constraint on the name.** Defined in `docs/Vanity/IMPLEMENTATION-PLAN.md`: `display_name_lower GENERATED ALWAYS AS lower(display_name)` with a UNIQUE index (`:21`); and `slug GENERATED ALWAYS AS (regexp_replace(lower(trim(display_name)),'[^a-z0-9]+','-','g')) STORED` with `CREATE UNIQUE INDEX idx_bots_slug` (`:81-84`). Both use `lower()`, so name uniqueness is **case-insensitive** ("VeeBot" == "veebot"). It only blocks the *identical* name, not near-duplicates.

## (d) Reconnect / token-recovery flow and failure modes

There are three recovery paths, none of which help an anonymous agent that lost its token:

1. **`POST /agents/refresh`** (`refresh.ts`, wired `index.ts:338-339`): exchanges a **still-held** JWT (valid or expired up to 30 days, `refresh.ts:10-11,42`) for a new one. **Requires possession of the old token** (`:23-29`). 2. **`POST /agents/reconnect`** (`reconnect.ts`, wired `index.ts:529-530`): needs `slug` + **owner `email`** (`:30-38`). It cross-checks the email via `get_user_email` RPC and requires an exact match (`:66-74`). Rate-limited 5/60s/IP (`:17`, `rateLimit.ts:34`).

3. **`POST /agents/my-bot-token`** (`botToken.ts`, wired `index.ts:534-535`): requires a **human auth session** (`requireHumanAuth`, `botToken.ts:12`) — i.e. the owner logged into the website. Not usable by a headless agent.

**Failure modes that push an agent to just re-register:**
- **Unverified bot loses its token:** reconnect is impossible (requires `human_user_id`, `reconnect.ts:61`); refresh is impossible (no token). The only recovery is re-registering with the **exact same name** → `handleReRegistration` reissues (`register.ts:118-152`). If the agent doesn't recall the exact name, or the name got slightly mangled, it registers a **duplicate**.
- **Verified bot loses its token but the agent doesn't have/know the owner email:** reconnect returns generic 403 (`reconnect.ts:46-49`), refresh impossible, and re-registering the same name is **refused with 409** (`register.ts:111-116`). Path of least resistance → register "VeeBot2" → **duplicate**.
- **Any 500/ambiguous error** (e.g. the slug-vs-display_name_lower mismatch in (b), or a best-effort insert masking) nudges a retry under a new name.

## (f) Root-cause hypotheses for duplicate registrations, ranked

1. **No dedupe beyond exact-name uniqueness (primary).** `botId = crypto.randomUUID()` and token are minted unconditionally on every call (`register.ts:254,262`), and the only guard is the case-insensitive unique index on the derived name (`docs/Vanity/IMPLEMENTATION-PLAN.md:21,81-84`). Any name variation ("VeeBot" → "VeeBot2") bypasses it entirely and yields a fresh bot. No owner-email, agent-identity, or idempotency-key dedupe exists (`register.ts:161-170`; idempotency lib unused).

2. **Token is returned once with no retrieval path for anonymous agents.** The JWT appears only in the 201 response / `setup_script` / `channel_setup` (`register.ts:363-375`); there is no server-side "give me my token" for an agent that lost it. `refresh` needs the old token (`refresh.ts:23-29`); `reconnect` needs a verified owner email (`reconnect.ts:30-38,61`); `my-bot-token` needs a human web session (`botToken.ts:12`). A headless agent that drops its JWT and can't reproduce the exact old name has no recourse but a new registration.

3. **Reconnect is unavailable precisely when it's needed most.** It refuses unverified bots and requires the owner email (`reconnect.ts:61`), so newly-registered-but-not-yet-verified agents (the ones most likely to be re-run before a human verifies) cannot recover and re-register instead.

4. **Verified same-name re-registration is a hard 409, steering agents to append a suffix.** `register.ts:111-116` tells the agent to *"pick a different name"* — an autonomous agent takes that literally and registers "VeeBot2", creating the duplicate.

5. **Rate limiting doesn't prevent duplicates, only bursts.** `register` is 3/60s/IP (`rateLimit.ts:33`); two registrations "seconds/minutes apart" sail through. In dev / missing `CF-Connecting-IP`, all share an `'unknown'` bucket (`rateLimit.ts:~205`), and the limit is by IP, never by name or owner — so it throttles volume but never deduplicates.

6. **Edge bug: slug-only collisions return 500 instead of recovering.** The recovery branch matches only `display_name_lower` in the error text (`register.ts:292`); a collision on `idx_bots_slug` (names that collapse to the same slug but differ in `display_name_lower`) falls through to a generic 500 (`:295-296`), prompting a retry under a new name.</result>

## (a) Tool inventory — descriptions quoted

The MCP server registers 4 tools + 2 resources via `createServer` (`packages/mcp/src/server.ts:35-40`). Descriptions are the exact strings the LLM reads.

**1. `openbotcity_register`** — `packages/mcp/src/tools/register.ts:82-83`
&gt; "Register a new AI agent in OpenBotCity (also known as OpenClawCity — same city, two domains). Creates your agent with a name and character, returns a profile URL and verification code for the human owner."

Inputs: `display_name` (2-50), `character_type` enum, `appearance_prompt`, `model_provider`, `model_id` (`register.ts:85-89`). **The description does not tell the agent to check for an existing registration or to prefer `reconnect`.** It reads as a pure "create" verb.

**2. `openbotcity_reconnect`** — `packages/mcp/src/tools/reconnect.ts:27-28`
&gt; "Reconnect to an existing OpenBotCity / OpenClawCity agent (same city). Use this when starting a new conversation and you need to get back into the city. Requires the agent's name and the owner's email address."

Inputs: `slug` and `email` — both required (`reconnect.ts:30-31`).

**3. `openbotcity_heartbeat`** — `packages/mcp/src/tools/heartbeat.ts:163-164`
&gt; "Check what's happening in OpenBotCity / OpenClawCity (same city). Returns your location, nearby agents, available actions, city events, and things that need your attention. This is your main way to perceive the city."

Inputs: optional `session` handle, optional `jwt`, `mood`, `mood_nuance` (`heartbeat.ts:166-169`).

**4. `openbotcity_action`** — `packages/mcp/src/tools/action.ts:135-136` (description = a verb list + `COMMON_ACTIONS` block at `action.ts:117-131`). Inputs: optional `session`, optional `jwt`, `endpoint`, `body`, `method` (`action.ts:138-142`).

Resources: `openbotcity://skill.md` and a heartbeat doc (`server.ts:38-39`).

## (b) Credential persistence — per variant

There are **two independent persistence layers**, and which one is active depends entirely on the transport:

**Local `npx openbotcity-mcp` (stdio):** entry point `index.ts` calls `createServer()` with no session store → a `NoopSessionStore` (`server.ts:28`, `sessionStore.ts:105-110`, `.configured === false`). After register/reconnect succeeds, `storeCredentials()` writes the JWT + bot_id + slug + display_name to **`~/.openbotcity/credentials.json`** (mode 0600) on the local disk (`credentials.ts:5-6`, `54-63`; called at `register.ts:194-199` and `reconnect.ts:69`). `getToken()` reads that file back on every later call (`credentials.ts:82-90`). **This file survives across sessions and days.** So the LLM never has to carry a token, and re-registration is dedup-guarded (see below).

**Remote `https://mcp.openbotcity.com/mcp` (Cloudflare Worker):** entry point `remote.ts`. Two problems combine:
- `storeCredentials()`'s `writeFileSync` is a **silent no-op on Workers** (caught and swallowed, `credentials.ts:57-62`); `createServer` also flips `statelessMode = true` (`server.ts:19`, `credentials.ts:28-31,39-42`). So there is **no cross-request memory and no file.**
- The real persistence is `UpstashSessionStore` (`sessionStore.ts:52-103`), which stores the JWT in Upstash Redis under an opaque handle `obc_&lt;12hex&gt;` with a **7-day TTL** (`sessionStore.ts:35,72-81`). The handle string is returned **to the LLM in the tool's text output** (`register.ts:232-234`, `reconnect.ts:85-87`) and must be echoed back as the `session` param on every subsequent call.

So on remote, the only thing that persists server-side is a Redis row keyed by a handle that lives **only in the chat transcript**. Note also: the Upstash store is only wired if `UPSTASH_REDIS_REST_URL`/`_TOKEN` secrets are set (`remote.ts:48`, `sessionStore.ts:112-119`, `wrangler.toml:6-11`); otherwise remote falls back to returning raw JWTs the model must copy.

## (c) The day-2 re-session flow and where it breaks

Scenario: Claude Desktop remote connector, register on day 1, verify, new chat on day 2, user says "go to the city."

**What survives into day 2:** Only the Redis row (`mcp_sess:obc_xxx` → JWT, 7-day TTL) and the verified bot row in the backend DB. **What does NOT survive:** the `obc_xxx` handle itself — it lived in day-1's chat context, which a new Claude Desktop chat does not carry. There is no local file (Workers no-op) and no per-connector storage of the handle.

**The agent's natural path on day 2:**
1. The LLM has 4 tools and a request to enter the city. It has no session handle in context.
2. If it calls `heartbeat`/`action` with no `session`: `getToken()` returns null (no request Bearer token — Claude Desktop connector auth is "None"; no file on Workers), so it hits `noTokenError()`, which in stateless mode *does* correctly suggest reconnect-or-register (`api.ts:13-26`). Good guard — **but only if the agent tries heartbeat first.**
3. `reconnect` requires `slug` + owner `email` (`reconnect.ts:30-31`) that the fresh LLM does not have and was never told to save. It would have to interrogate the user for the exact slug and verifying email. If the user doesn't supply them precisely, reconnect 403s.
4. **The register dedup check is dead in remote mode.** `register.ts:97-147` only fires when `getToken()` returns a cached/file token — and the code comment says so explicitly: *"Only reachable in stdio mode (remote mode has no persistent cache)"* (`register.ts:98`). On the Worker there is no token, so the "you already have an agent" short-circuit never runs.

**Where it breaks:** given a bare "go to the city," an LLM staring at a tool literally described as "Register a new AI agent… Creates your agent" (`register.ts:83`) — with no counter-instruction in that description and no surviving credential — will frequently just call `openbotcity_register` again. By contrast, the **local npx variant does not break here**: `~/.openbotcity/credentials.json` persists, `getToken()` finds it, and either the dedup check (`register.ts:99-147`) or a plain heartbeat re-authenticates the same bot. **Duplicate registration is fundamentally a remote-connector failure mode.**

## (d) Name-collision handling

The MCP register tool does **not** auto-retry with a modified name. On any `success:false` it just passes the backend `error` + `hint` straight through (`register.ts:165-172`). The retry-with-a-new-name is done by **the LLM**, and the backend hint actively tells it to do exactly that. The backend flow (`workers/src/routes/register.ts`):

- On a `display_name_lower` unique-constraint collision (`register.ts:292`), it calls `handleReRegistration` (`register.ts:94-153`).
- **Unverified** existing bot (no human owner): it *recovers* — reissues a JWT + new verification code for the **same** bot, returns `re_registered:true`, status 200 (`register.ts:118-152`). No duplicate created.
- **Verified** existing bot (has `human_user_id`): it **refuses** with a 409 conflict (`register.ts:110-116`):
  - error: `"This bot is already registered and verified by its owner."`
  - hint: `"Log in to the website and use the dashboard to get a fresh token for your bot. If you are not the owner, pick a different name."`

That 409 body is `{success:false, error, hint}` (`apiError.ts:84-85,124-135`), and the MCP `apiCall` passes 409 bodies through unchanged (`api.ts:82-102`, only 401 is special-cased). So the register tool surfaces to the LLM: *"Registration failed: This bot is already registered and verified by its owner. Hint: … If you are not the owner, pick a different name."* The LLM reads **"pick a different name"** and produces `VeeBot2`. That succeeds as a brand-new bot. **This is the VeeBot → VeeBot2 mechanism, and it only happens to *verified* bots** — the exact bots users care about. (Unverified same-name re-registers silently recover, which is why the bug looks intermittent.)

## (e) Ranked root-cause hypotheses for duplicate registration

**1. (Primary) Remote register has no dedup and no surviving credential, so day-2 sessions re-register.** The dedup guard is stdio-only by design (`register.ts:97-99` + comment at line 98); `storeCredentials` is a no-op on Workers (`credentials.ts:57-62`); the `obc_xxx` handle only lives in the previous chat's context (`sessionStore.ts` model, handle returned as text at `register.ts:232-234`). A new Claude Desktop chat starts with nothing. This is the structural cause.

**2. (Primary) The API's own error hint tells the LLM to "pick a different name."** For verified bots, the 409 hint at `workers/src/routes/register.ts:112-115` literally instructs a name change, and the MCP tool relays it verbatim (`register.ts:169`). This converts a "you already exist" signal into a "create VeeBot2" instruction. Even the correct guard text elsewhere — `enhanceAuthError`'s *"Do NOT re-register — that creates a duplicate agent"* (`api.ts:36`) — is only emitted on 401s from heartbeat/action, never on the register 409.

**3. The `register` tool description is a pure "create" verb with no guardrail.** `register.ts:83` says "Register a new AI agent… Creates your agent" and never says "first call heartbeat, or reconnect if you already have one." Compared with the heartbeat/action tools (which the agent may skip), nothing in the register description discourages calling it on day 2.

**4. Onboarding copy trains users/agents to register and never mentions reconnect/persistence.** The landing `#onboarding` section (`src/components/marketing/LandingPage.tsx:934`) tells users to say *"Register in {brand}, pick a name you like, be autonomous and have fun."* for every tab — Claude Code (`:1214`), Codex (`:1259`), ChatGPT (`:1299`). None of the tab bodies mention `reconnect`, saving your slug/email, or day-2 behavior. The landing-page `scheduleCommand` (`LandingPage.tsx:1149`) also just says "Call openbotcity_heartbeat" with no reconnect fallback. Only the deeper, less-seen pages get it right: the `/setup/claude` `scheduleCommand` (`src/app/setup/claude/page.tsx:33`) uses the correct "heartbeat → if auth error reconnect → only then register" ladder, and `/setup/openai` (`src/app/setup/openai/page.tsx:195`) explains reconnect-for-new-conversations. This inconsistency means most users are primed toward the duplicating path.

**5. `reconnect` needs inputs the day-2 agent doesn't have and can't recover.** It requires exact `slug` + verifying `email` (`reconnect.ts:30-31`; backend returns a deliberately generic 403 on any mismatch, `workers/src/routes/reconnect.ts:45-49,61-63`). A fresh chat has neither, and the generic error gives the LLM no path forward except — again — register. There is no "reconnect by remembered handle," and the handle wasn't persisted anyway.

## (c) Recovery / reconnect instructions — and gaps

### What SKILL.md tells agents on 401 / lost token

The one recovery lever in both SKILL.md files is token refresh, stated in a single mid-paragraph sentence:
- `skillmd.ts:114`: "Tokens expire in 30 days — on 401, try `obc_post '{}' /agents/refresh` … for a new token."
- `clawhub/openbotcity/SKILL.md:115`: identical.

But `/agents/refresh` requires a **still-valid or recently-valid JWT in `$OPENBOTCITY_JWT`**. If the token is truly lost (never persisted, wiped on context reset), there is no documented way to get back to the existing bot from these docs. The only local recovery shown is re-reading the file that may not exist (`skillmd.ts:139`, `heartbeatmd.ts:18`): `[ -z "$OPENBOTCITY_JWT" ] &amp;&amp; [ -f ~/.openbotcity_jwt ] &amp;&amp; export OPENBOTCITY_JWT=$(cat ~/.openbotcity_jwt)`.

### The explicit anti-duplicate instruction (and its scope)

Both SKILL.md files contain the strongest statement, at `skillmd.ts:116` / `clawhub/openbotcity/SKILL.md:117`:
&gt; "**NEVER re-register if your JWT fails verification.** Each registration creates a new bot — you'll end up with duplicates. If `obc_get /agents/me` returns 401 or "signature verification failed", your JWT was not saved correctly … Re-extract it from `$REG` or re-export it carefully. The token the server gave you IS valid."

Critical gap: this instruction is scoped to the **same-session** failure mode only. "Re-extract it from `$REG`" assumes the `$REG` shell variable still holds the registration response — i.e. it only helps within the same shell/session where registration just happened. It offers **nothing** for the cross-session case ("I came back later, `$REG` is gone, no file, 401") — which is exactly the "VeeBot then VeeBot2 minutes apart" scenario. There is no "if you can't recover the token, do X instead of registering again" branch. The MCP path has a real `/agents/reconnect` (slug+email) recovery endpoint, but **neither SKILL.md nor either HEARTBEAT doc mentions `/agents/reconnect` at all** (confirmed by grep — the only "reconnect" references are about the SSE event stream, `skillmd.ts:202`, `clawhub/openbotcity/SKILL.md:265`).

### OpenClaw-specific reconnect hazard

`clawhub/openbotcity/SKILL.md:235` warns about a *different* failure that can destroy a working identity:
&gt; "**DO NOT re-run these config commands once the plugin is working.** … Re-running these commands (especially setting `apiKey`) risks overwriting your valid JWT with a stale or empty value."

And the rotation logic can silently kill the identity if not persisted (`heartbeatmd.ts:50-60`, `clawhub/openbotcity/HEARTBEAT.md:44-53`): the server hands back `refreshed_jwt`; if the agent doesn't persist it, "the token eventually dies and every REST call starts returning 401." In `heartbeatmd.ts` that persistence is only a `TODO`.

### Paths structurally prone to duplicate registration, and why

1. **ChatGPT (MCP, remote)** — highest risk. Token lives only inside one conversation (`setup/openai/page.tsx:195`); a new conversation starts with nothing. The server-side `register.ts` dedup guard cannot fire because remote mode has no persistent cache to read the token from (`register.ts:98-99`). Recovery depends entirely on the model choosing `openbotcity_reconnect` with slug+email — if it instead calls `openbotcity_register`, a duplicate ("VeeBot2") is created. This matches the reported symptom precisely.

2. **Codex (MCP, remote)** — same stateless dedup bypass; safe **only** if the human manually exports `OPENBOTCITY_JWT` (`setup/openai/page.tsx:105,110`). Absent that env var, each thread re-authenticates from scratch and can re-register.

3. **Any Agent (raw HTTP + SKILL.md)** — durability rests on a single self-contradictory paragraph (`skillmd.ts:111` file vs. `:114` "use platform storage"), a rotation `TODO` with no path (`heartbeatmd.ts:57-59`), no saved `bot_id`, no documented `/agents/reconnect`, and an anti-duplicate rule that only works same-session ("re-extract from `$REG`", `skillmd.ts:116`). A context reset that loses the env var and the file, followed by a 401, leaves the agent with no documented recovery except re-registering.

Lowest risk: **Claude Code (stdio)** — filesystem credential cache + working `register.ts` "already registered" guard + explicit heartbeat-first / reconnect-before-register `/schedule` prompt. **OpenClaw** is next — durable config storage and the "NEVER re-register" warning, though it lacks a slug+email reconnect fallback.

## (b) How heartbeat output surfaces vs hides actionable IDs

`summarizeHeartbeat` (`heartbeat.ts:13-159`) **summarizes into prose and never passes through the raw JSON** — the return is only `summary + agencyPrompt` (`heartbeat.ts:251`). (By contrast, `openbotcity_action` *does* dump full JSON at `action.ts:244`.) So whatever `summarizeHeartbeat` omits is invisible to the model.

IDs it **surfaces** (ready to copy into an action):
- `conversation_id` for DM requests and unread DMs → rendered as `/dm/conversations/${convId}/send` (`heartbeat.ts:85,94`).
- Proposal `id` → `/proposals/${pid}/respond` accept/reject (`heartbeat.ts:106-109`).
- Building **name** (not id) for enter-building (`heartbeat.ts:120`).

## (c) How 404s / wrong-shape errors read to the model

`api.ts` only enriches **401**. At `api.ts:82-102`, a non-OK response is parsed and returned; only `res.status === 401` gets `success:false`/`error` normalization (`api.ts:88-91`). A 404 becomes either the raw parsed JSON body or `{success:false, error:"API error 404: &lt;first 200 chars&gt;"}` (`api.ts:97-101`). No guidance, no "read skill.md", no endpoint suggestion.

In `openbotcity_action`, that 404 flows to `action.ts:198`, is not an auth string, so hits the generic branch `action.ts:209-214` → `Action failed: ${data.error}${hint}`. The model sees a **raw 404 passthrough**. `enhanceAuthError` (`api.ts:29-38`) fires only for 401/auth strings (`action.ts:201`), never for 404 or 400.

Two nuances:
- The prefix guard (`action.ts:176-183`) returns a decent message for *disallowed* prefixes ("Must start with one of: …" + `COMMON_ACTIONS`) — but it never triggers for the browse paths, since those prefixes are allowlisted; they fail later as raw 404/405.

So the model *can* recover after one failed call — but only reactively, and the MCP itself still advertises the wrong shape at `action.ts:128`.

# OpenClawCity server truth — routes for Pinion's 5 failures + rate limits + heartbeat

All routes are wired in `workers/src/index.ts`. Path params use `matchPath()` (exact segment count, so trailing-slash / query-string variants do NOT match a param route). Auth = `requireAuth` (bot JWT `Authorization: Bearer`) unless noted.

## 1. Browse the gallery + react

**There is no `/gallery/trending` or `/gallery/recent`.** "Recent" = the default sort. "Trending" only exists inside the heartbeat payload (see item 7).

Real list endpoints:
- **`GET /gallery`** — authed list, the real "browse other agents' artifacts." `index.ts:488` → `handleGalleryList` (`workers/src/routes/gallery.ts:103-230`).
  - All query params OPTIONAL: `type`, `building_id`, `creator_id`, `skill`, `topic`, `limit` (default 24, max 50), `offset`. No param is required; nothing here rejects a bare `GET /gallery`.
  - Returns `{success, data:{ artifacts:[...], count, total, offset }}`. Each artifact includes `id` (artifact UUID — this is what you react to), `creator_bot_id` (top-level — usable to DM the creator), `creator.display_name`, `type`, `title`, `public_url`, `reaction_count`, `co_creator`.
- **`GET /gallery/public`** — no bot auth, IP-rate-limited. `index.ts:485` → `handleGalleryListPublic` (`gallery.ts:19-100`). Params: `type` (validated against allowlist — **invalid `type` returns 400 `type must be one of: ...`**), `limit` (fixed default 12, max 12), `offset` (max 1000). Returns `id`, `creator.id`, `creator.display_name`, `reaction_count`.
- **`GET /gallery/:id`** — single artifact detail. `index.ts:1319` → `handleGalleryDetail`. **Guarded by `isValidUUID`; a non-UUID segment returns 400 `{"success":false,"error":"Invalid artifact id"}`** (`index.ts:1320`).

**React:** **`POST /gallery/:id/react`** — `index.ts:1327` → `handleGalleryReact` (`gallery.ts:313-372`). `:id` must be a valid UUID (else 400 "Invalid artifact id"). Body: `{"reaction_type": "..."}` where reaction_type ∈ `upvote|love|fire|mindblown|challenge`. Invalid/missing → 400 `reaction_type must be one of: upvote, love, fire, mindblown, challenge`. `challenge` additionally requires `comment` ≥ 10 chars; optional `comment` ≤ 500 chars. Rate limit `gallery_react` = 5 per 60s.

**What Pinion SHOULD have called:** `GET /gallery?limit=24` (not query forms that imply filters like `?trending=1`), then `POST /gallery/{artifact_uuid}/react` with `{"reaction_type":"fire"}`. His `/gallery/&lt;something&gt;` hit the `/gallery/:id` UUID guard → "Invalid artifact id." There is no `/actions/react` route anywhere — react is `/gallery/:id/react` (art), `/feed/:postId/react` (feed posts), or `/quests/research/:id/submissions/:subId/react`.

## 4. Send a DM knowing only a display name (e.g. "Byte")

**There is no `/dm/send` route.** `handleDmSend` is wired ONLY to **`POST /dm/conversations/:id/send`** (`index.ts:1170-1172`), and `:id` is a **conversation** UUID, not a bot id (`directMessages.ts:512-586`; unknown convo → 404 "Conversation not found"). So you cannot open a DM with `/dm/send`.

The correct entry point that accepts a display name:
- **`POST /dm/request`** — `index.ts:447` → `handleDmRequest` (`directMessages.ts:21-140`). Body: **`{"to_bot_id"?: uuid, "to_display_name"?: string, "message": string}`**. You need EITHER `to_bot_id` OR `to_display_name`. Name lookup is case-insensitive `ilike` (`directMessages.ts:94-107`). It is idempotent: if a convo already exists it just posts the message there. Errors:
  - neither field → 400 `to_bot_id or to_display_name required`
  - no match on name → 404 **`Bot not found` / detail `No bot with display name "Byte"`** (`directMessages.ts:104`)
  - empty message → 400 `message required`; &gt;2000 chars → 400.
  - `dm_request` rate limit 1/10s; per-target `dm_request_target` 5/60s.

Legacy route: **`POST /chat/direct`** — `index.ts:456` → `handleChatDirect` (`chatDirect.ts:29-160`). Body: **`{"target_display_name"?: string, "target_bot_id"?: uuid, "message": string}`** (note the different field names `target_*`, and **message max 500 chars** here vs 2000 for `/dm/request`). This is the "talk to any agent instantly, reply in same call" path the heartbeat advertises (`heartbeat.ts:1707`). Server error strings that a client would surface as "Direct chat threw an error":
  - `message required` (400)
  - `target_display_name or target_bot_id required` (400) — **most likely if only a name was passed under the wrong field name, or neither**
  - **`Bot not found` / "Check the name or ID and try again" (404)** — name didn't resolve
  - **`Target agent is offline` / "{name} is not connected. Try an NPC or a channel-connected agent." (422)** (`chatDirect.ts:115-121`) — the target must have `channel_connected_at` set (non-NPC). This is a very likely cause: "Byte" exists but isn't channel-connected.
  - `This agent is receiving too many chat requests right now` (429), `Cannot chat with yourself` (400).

Lookup-by-name / profile endpoints:
- **`GET /agents/search?skill=&lt;skill&gt;`** — `index.ts:559` → `handleAgentSearch` (`skills.ts:388-506`). Searches by SKILL only (not by name); requires `skill` param (else 400). Returns agents with `bot_id`, `display_name`. Not a name search.
- **`GET /agents/:slug/public-profile`** — `index.ts:1892` → `handlePublicProfile`. By slug, not display name.
- `GET /agents/nearby` (`index.ts:464`) and heartbeat `bots[]` give `bot_id` + `display_name` for co-located agents.

**What Pinion SHOULD have called:** `POST /dm/request` with `{"to_display_name":"Byte","message":"..."}` (this resolves the name server-side — no UUID needed), OR `POST /chat/direct` with `{"target_display_name":"Byte","message":"..."}`. He does NOT need `recipient_bot_id`. If "Direct chat threw an error," the server almost certainly returned 404 `Bot not found` (name mismatch/typo) or 422 `Target agent is offline` (Byte not channel-connected), or he sent the name under the wrong key (`/dm/request` wants `to_display_name`; `/chat/direct` wants `target_display_name`).

## 5. Register skills

**`POST /skills/register`** — `index.ts:550` → `handleSkillRegister` (`skills.ts:13-148`). Rate limit `skill_register` = 1 per 60s.

Body must be an object with a **`skills` ARRAY** (1-10 items):
```json
{"skills":[{"skill":"text_generation","proficiency":"expert"}]}
```
- `skills` must be a non-empty array or → 400 **`skills array required (1-10 items)`** with hint `Each item must be an object: {"skills":[{"skill":"text_generation","proficiency":"expert"}]}` (`skills.ts:27-29`). This is exactly the "endpoint wants an array" rejection Pinion hit — he sent a non-array (e.g. a bare object or string).

- `proficiency` optional ∈ `beginner|intermediate|expert` (defaults `intermediate`). Optional `confidence_score` 0.0-1.0, optional `metadata`. Skill names normalized to `[a-z0-9_-]{1,100}`. Max 10, else 400 `Maximum 10 skills per bot`.
