# Implementation Plan: 3D Interior for The Byte Cafe

## 1. Architecture Overview

### The Core Idea: Phaser Sleeps, R3F Wakes

The current flow is:
- `WorldScene` (Phaser 2D) detects building entry -> emits `scene-changed` event -> starts `BuildingInteriorScene` (Phaser 2D)

The new flow for the Byte Cafe:
- `WorldScene` detects Byte Cafe entry -> emits `scene-changed` with `{ scene: 'building-3d', buildingType: 'cafe', ... }` -> Phaser pauses (does NOT start BuildingInteriorScene) -> React mounts R3F `<ByteCafe3D>` component over the Phaser canvas
- Exit: R3F component unmounts, Phaser resumes `WorldScene`

This is architecturally clean because the existing React HUD layer in `src/app/page.tsx` already manages scene state via `handleSceneChanged`. The R3F canvas is a sibling to the Phaser `<div>`, conditionally rendered when `currentBuildingType === 'cafe'` and the scene is `'building-3d'`.

### Why Not Embed R3F Inside Phaser?

Phaser and Three.js both own a WebGL context. They cannot share one. The correct approach is to **hide** the Phaser canvas (or pause+fade) and **mount** a separate R3F `<Canvas>` at the same DOM position. This is exactly how Claw3D works -- it renders R3F as a full React component, not inside a game engine.

- Each agent is a `<Billboard>` from drei containing a textured plane
- The texture is the agent's existing sprite sheet frame (idle/walking/action)
- Speech bubbles render as `<Text>` components above the billboard
- Name labels below

This approach:
- Reuses all 704 existing PixelLab PNGs
- Maintains visual consistency with the 2D world
- Is dramatically simpler than Claw3D's procedural body parts
- Looks charming -- pixel sprites in a 3D room is an established aesthetic (Octopath Traveler style)

- **Billboard sprites** are far cheaper than procedural 3D characters (1 quad vs 20+ meshes per agent)
